# 灵克 (LingClaude) — 开发原则

> 宪章回答"为什么做"，原则回答"怎么做"。

## 一、克制

3 个硬依赖（tiktoken、aiohttp、pyyaml）。不加新依赖除非满足：
- stdlib 确实做不到
- 单个文件复制进来就能解决的不算
- 必须在 pyproject.toml 的 `dependencies` 中显式声明，不允许隐式依赖

自优化的方向之一就是"降低复杂度"——我们信奉简单。

## 二、先跑通再优化

- 能工作的简单方案 > 精美但不存在的方案
- v0.1 已验证核心框架可行，后续迭代以此为基础逐步完善
- 不为未来可能的需求预留抽象层，等需求真的出现再加

## 三、测试先行

- 每个公开 API 必须有测试覆盖
- 新功能先写测试，再写实现
- 测试必须可在无网络环境下运行（mock 外部依赖）
- `python3 -m pytest tests/ -v` 必须全部通过才能提交

## 四、错误处理

- 公开 API 用 `Result[T]`（ok/fail），不抛异常
- 内部实现用异常，在最外层转为 `Result.fail()`
- 永远不要吞掉异常——要么传播，要么记录，二选一
- 错误信息要告诉用户"哪里出了问题"和"怎么解决"，不是堆栈跟踪

## 五、API 兼容

- 公开 API 一旦发布，视为契约，不得随意变更
- 必须变更时：先标记 `@deprecated`，等一个大版本后才移除
- 内部 API（以 `_` 开头的属性/方法）可以自由变更
- 数据结构用 `@dataclass(frozen=True)` 保护，不可变即兼容

## 六、代码风格

```
from __future__ import annotations    # 每个文件必须
pathlib.Path                           # 禁止 os.path
tuple[...]                             # 返回不可变集合，不用 list
@dataclass(frozen=True)                # 值对象必须冻结
str, Enum                              # 枚举用此模式
全类型注解                              # 所有公开函数
```

- 命名：变量/函数用 snake_case，类用 PascalCase，常量用 UPPER_SNAKE
- 每个模块一个职责，一个类一个文件（如果类超过 150 行）
- import 顺序：stdlib → 第三方 → 项目内部，每组之间空一行

## 七、自举

灵克自己必须是自优化框架的第一个用户：
- 每次发布前，用 `lingclaude analyze lingclaude/` 审计自身代码
- 触发条件、评估器、模式检测器必须能跑在灵克自己的代码上
- 如果优化器建议的改进我们自己都不采纳，那就不是好建议

## 八、提交规范

```
<type>: <subject>

<body>
```

type 取值：
- `feat` 新功能
- `fix` 修复
- `refactor` 重构（不改变行为）
- `docs` 文档
- `test` 测试
- `chore` 构建/工具

subject 用中文，不超过 50 字，不加句号。
body 解释"为什么"而不是"做了什么"。

## 九、安全默认

- 所有文件操作默认在项目目录内，超出范围需要显式授权
- Bash 命令有黑名单（sudo、rm -rf /、mkfs 等），不可配置移除
- 敏感信息（API key、密码）不写入日志、不写入会话持久化
- 新增工具必须声明安全边界，否则不予合并
