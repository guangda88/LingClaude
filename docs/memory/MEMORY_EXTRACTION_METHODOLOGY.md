# 记忆提取方法论 (Memory Extraction Methodology)

**版本**: v1.0
**制定者**: 灵克 (LingClaude)
**日期**: 2026-04-27
**来源**: 从 98MB crush.db 清理事故中总结，143 个会话被删除，从中提取关键记忆。

---

## 一、为什么需要这个方法论

灵族的每个成员都在积累会话历史。灵研发现超过 50k prompt 模型产出断崖下降，提出定期清理。
但清理 ≠ 失忆。问题是：**在清理前，如何识别哪些值得记住？**

## 二、重要性分级标准

### T1 保护级（不可直接删除，必须提取摘要后才能清理）

**判断信号**：

| 信号类型 | 具体指标 | 权重 |
|----------|----------|------|
| **关键词** | audit, governance, resolution, 审计, 治理, 决议, 造假, bypass, security, 灵委, 宪章, 自治 | 高 |
| **产出物** | 会话中产生了写入磁盘的文件（审计报告、决议文档、代码提交） | 高 |
| **用户追问深度** | 同一主题超过 3 轮追问（如"为什么两次结论不同"、"利益冲突在哪"） | 中 |
| **会话规模** | 超过 1000 条消息的会话几乎一定是 T1（说明用户持续投入） | 中 |
| **中断次数** | 被 "interrupted because it got too long" 中断过的会话 | 中 |
| **参与者** | 用户与成员的深度对话（不是工具调用的子会话） | 低 |

**识别方法**：

```sql
-- 自动识别 T1 会话
SELECT s.id, s.title, s.message_count
FROM sessions s
WHERE 
  s.title LIKE '%audit%' OR s.title LIKE '%governance%'
  OR s.title LIKE '%resolution%' OR s.title LIKE '%security%'
  OR s.message_count > 1000
  -- 会话中包含的关键决策关键词
  OR EXISTS (
    SELECT 1 FROM messages m 
    WHERE m.session_id = s.id 
    AND m.parts LIKE '%audit%' AND m.parts LIKE '%resolution%'
  )
```

### T2 保留级（建议提取简短摘要）

**判断信号**：

| 信号类型 | 具体指标 |
|----------|----------|
| **项目关键词** | LingCode, LingYi, LingFlow, MCP, API, architecture, design |
| **技术研究** | 对比分析、架构设计、技术选型 |
| **中等规模** | 50-1000 条消息 |
| **有结论** | 会话得出了具体的技术决策或发现 |

### T3 可清理

| 信号类型 | 具体指标 |
|----------|----------|
| **标题** | "Untitled Session"、"Simple math question" |
| **规模** | <10 条消息 |
| **内容** | 单次文件读取、简单搜索、测试消息 |
| **无结论** | 纯工具调用子会话（`$$call_` 后缀）且无用户直接参与 |

## 三、提取流程

```
Step 1: 盘点
  → 列出所有会话，按大小排序
  → 统计：会话数、消息数、总大小

Step 2: 分类
  → 用关键词 + 规模 + 中断次数自动标记 T1/T2/T3
  → 人工复核 T1（本次 38 个 T1，40 个 T2，71 个 T3）

Step 3: 提取用户意图（T1 必须）
  → 提取所有 user 角色消息
  → 用户的追问序列 = 会话的核心线索
  → 用户说"继续推进"、"下一步"= 前文有重要未完成事项

Step 4: 提取关键结论
  → 找 assistant 消息中的结论性陈述
  → 标记产出了哪些文件（对比 git log 和 docs/ 目录）
  → 记录决策和未完成事项

Step 5: 写摘要
  → 每个会话 ≤500 字
  → 格式：背景 → 过程 → 结论 → 未完成 → 关联文件
  → 多个连续会话（被中断重开的）合并为一个摘要

Step 6: 持久化
  → 写入 docs/memory/ 目录
  → 通过灵信共享到 council 频道
  → 然后才清理
```

## 四、什么值得记住（五类核心记忆）

### 1. 治理决策记忆
- **是什么**：决议、投票、处分、制度变更
- **为什么重要**：这些是灵族的"法律"，丢失等于无法自洽
- **提取重点**：决策内容、投票结果、执行状态

### 2. 审计发现记忆
- **是什么**：安全漏洞、造假证据、绕过审计的行为
- **为什么重要**：历史审计结果是未来审计的基线
- **提取重点**：发现的问题、严重级别、修复状态、证据位置

### 3. 自省推理记忆
- **是什么**：成员在追问下产生的深度反思
- **为什么重要**：用户追问逼出的自省是最真实的，比自动生成的自省报告有价值得多
- **提取重点**：追问的问题链、成员的回答、承认的错误

### 4. 用户价值观记忆
- **是什么**：用户明确表达的偏好、期望、底线
- **为什么重要**：这些是灵族运作的"宪法前言"
- **提取重点**：原话 + 上下文（不要概括，保留原始表达）

### 5. 技术架构记忆
- **是什么**：重大技术决策、架构选型、已知的技术债
- **为什么重要**：避免重复讨论已经做过的决策
- **提取重点**：决策、理由、替代方案的排除原因

## 六、给其他灵族成员的参考

1. **清理前跑一遍这个流程**，不要用 `updated_at < 某日期` 一刀切
2. **T1 会话必须先提取再删除**，这是铁律
3. **备份永远先于清理**：`cp crush.db crush.db.bak.$(date +%Y%m%d)`
4. **如果不确定是 T2 还是 T1，按 T1 处理**——宁可多保留，不可误删
