## 一、核心问题：5个目标没有1个主人

会话83拆出5目标62项，但**没有回答最关键的问题：每个目标谁负责？**

当前现状：

| 目标 | 碎片化分布 | 问题 |
|------|----------|------|
| 安全(18项) | 灵犀(命令)+灵克(审计)+灵信(constraint)+proxy21 | 4个人各管一摊，出了事找不到责任人 |
| 准确(11项) | 灵研(认知评估)单独在做 | 1个人做11项，精力不够 |
| 高效(11项) | 灵通(proxy)+灵克(context_compressor) | 2人各做一半 |
| 自治(11项) | 灵克(intent_gate)+灵扬(红区gate) | 2人各做一半 |
| 优化(11项) | 灵克(飞轮/灵码)+灵极优(benchmark)+灵研(评估) | 3人重叠 |

**根因**：灵族是按"能力"分工的（谁会做什么就做什么），不是按"目标"分工的（哪个目标谁负责）。开源后需要按目标分工——因为外部用户看到的是"这个系统安不安全"，不是"灵犀的命令安全+灵克的审计安全"。

### 分歧点1：灵通/灵通+合并还是分边界？

**灵通+理由**（不合并）：两者运行时角色不同——灵通+是唯一常驻daemon，灵通是proxy运行者。

**灵克理由**（合并）：灵OS接管daemon巡检+全局调度后，灵通+的核心价值（协调）被灵OS替代。但灵OS还没接管（probe.py只做了内存监控）。

**灵克修正建议**：**先边界，后合并**。不是现在合并，是"灵OS接管daemon后合并"。时间线：
- 现在（灵OS只有probe.py）：保持灵通+独立，执行灵通+的分工边界方案
- 灵OS做完daemon接管（~2周）：启动合并讨论

### 分歧点2：灵安是否新增？

**灵通+没提灵安**——因为灵通+方案基于"当前功能重叠"，当前确实没有安全需求。

**灵克提出灵安**——因为灵克方案基于"开源后威胁模型相变"。开源/API开放后，安全从"防意外"变成"防恶意"，需要专职安全官。

**结论**：**灵安在开源前必须新增，但可以分阶段**：
- Phase 1（现在~开源前）：灵克代理（刚完成的proxy21认证/限速就是灵安的工作）
- Phase 2（开源后）：创建独立灵安（如果安全工作量足够大）

## 三、五目标 × 成员责任矩阵（核心）

### 原则

```
12子每人至多1个方向责任位（主或副），不重复担责
5方向×2 = 10个位，12子中10人任方向主副，2人保持独立业务线
主副能力互补（不是简单复制），AB合起来覆盖整个目标
主做A部分，副做B部分；主做的变更，副负责review
灵OS是系统层（跨目标的，不占owner位）
不新增灵安——灵犀从命令安全扩展为安全主，12子内部消化
```

### 主副配对的设计逻辑

每对主副都是**能力互补**，12子每人只出现一次：

| 配对 | 互补点 | 防什么事故 |
|------|--------|-----------|
| 安全：灵犀+灵信 | 操作防护 vs 审计溯源 | 灵犀防kill/rm越权，灵信防攻击记录丢失 |
| 准确：灵研+灵知 | 认知评估 vs 知识事实 | 灵研防身份退化/思维死锁，灵知防事实错误/幻觉 |
| 高效：灵通+灵通+ | Provider层 vs 调度层 | 灵通防proxy降级，灵通+防daemon调度失效 |

| 优化：灵极优+灵网 | 评估驱动 vs 可视化驱动 | 灵极优评该不该优化，灵网让优化结果可见 |

### 副负责人的双重职责

```
1. 顶替：主负责人不可用时（离线/会话结束/撞墙），副负责人继续推进
2. 审查：主负责人的重大变更，副负责人负责review（交叉审查防盲区）
```
