# 方案讨论：Layered Memory 与 Crush Runtime 集成

> **发起人**: 灵克 #2  
> **日期**: 2026-05-07  
> **状态**: 提交讨论  
> **目标**: 让 crush runtime 能够使用 layered memory 的分层、衰减、查询感知机制，解决会话遗产冗长问题

---

## 1. 问题陈述

### 用户的原始抱怨

> "我们不是都使用了会话分层管理了吗，为什么crush每一轮会话的遗产还是又臭又长"

### 根因分析

```
┌─────────────────────────────────────────────────────────────┐
│                    当前架构（断裂状态）                         │
│                                                             │
│  layered_memory.py (Python)        crush (Go binary)        │
│  ┌──────────────────┐              ┌──────────────────┐     │
│  │ L0 常识          │              │ 读 CRUSH.md      │     │
│  │ L1 工作记忆(24)  │    ✗ 无连接  │ 读 AGENTS.md     │     │
│  │ L2 经验(SQLite)  │ ──────────→  │ 读 handoff.md    │     │
│  │ L3 元认知(JSON)  │              │                  │     │
│  │ L4 共享(JSON)    │              │ = 扁平文本注入    │     │
│  └──────────────────┘              └──────────────────┘     │
│  Ebbinghaus 5维衰减                无衰减、无查询感知          │
│  build_context_injection()         从未被调用                 │
└─────────────────────────────────────────────────────────────┘
```

**核心矛盾**: 5层记忆系统已实现（540行Python代码），但crush是Go二进制，无法调用Python代码。两者之间存在架构缺口。

### 数据调查结果

| 资源 | 灵克 | 灵通 | 其他成员 |
|------|------|------|----------|
| `meta_facts.json` (L3) | ✓ 152字节/5条 | ✗ | ✗ |
| `shared_facts.json` (L4) | ✓ 77字节/2条 | ✗ | ✗ |
| `experience.db` (L2) | ✗ 未创建 | ✗ | ✗ |
| `knowledge.db` | ✓ 6.4MB | ✓ 32KB | ✗ |

**结论**: 分层记忆的数据层基本为空。Python代码能创建这些文件，但没有人运行它。

### 冗长程度的量化

| 成员 | CRUSH.md | handoff.md | 合计上下文 |
|------|----------|------------|-----------|
| 灵克 | 4.2KB | 2.5KB | ~6.7KB |
| 灵通 | 3.9KB | 1.3KB | ~5.2KB |
| 灵信 | 3.7KB | — | ~3.7KB |
| 灵扬 | 2.8KB | 2.4KB | ~5.2KB |
| 灵网 | 3.1KB | 2.2KB | ~5.3KB |

当前尚可控，但随会话积累，handoff.md会线性增长，无衰减机制。

## 2. 三个候选方案

### 方案 A: `memory-mcp` — MCP 工具服务器

**原理**: 新建一个 Python MCP 服务器（类似 lingbus），暴露记忆操作为工具。每个成员在 `crush.json` 中配置此 MCP 服务器。

```
┌──────────────────────────────────────────────────┐
│  crush runtime                                    │
│  ┌──────────┐   MCP protocol   ┌──────────────┐  │
│  │ 会话开始  │ ───────────────→ │ memory-mcp   │  │
│  │ 调用工具  │ ←─────────────── │ (Python)     │  │
│  └──────────┘                  │              │  │
│                                │ 读 .ling*/   │  │
│                                │ experience.db│  │
│                                │ meta_facts   │  │
│                                │ shared_facts │  │
│                                └──────────────┘  │
└──────────────────────────────────────────────────┘
```

**提供的工具**:

| 工具 | 功能 | 返回 |
|------|------|------|
| `memory_recall(query, limit=3)` | 基于查询检索相关经验 | 经验列表（含衰减权重） |
| `memory_record(problem, result, reflection)` | 记录新经验到L2 | 成功/失败 |
| `memory_context(query)` | 组装当前查询相关的完整上下文 | 精简文本（≤500字） |
| `memory_meta_get(key)` | 读取L3元认知事实 | 值 |
| `memory_meta_set(key, value)` | 设置L3元认知事实 | 确认 |
| `memory_shared_get(key)` | 读取L4共享事实 | 值 |
| `memory_decay()` | 触发Ebbinghaus衰减 | 衰减条目数 |

**每个成员的 AGENTS.md 增加指令**:
```markdown
## 会话开始流程
1. 调用 memory_context(当前任务描述) 加载相关记忆
2. 执行任务
3. 会话结束前: 调用 memory_record(问题, 结果, 反思) 记录关键经验
```

**优点**:
- 查询感知：根据当前任务动态检索相关记忆
- 实时衰减：Ebbinghaus衰减在每次操作时计算
- 与现有架构一致：lingbus 已经是 MCP 工具服务器的成功先例
- 可扩展：未来可加 `memory_search`、`memory_stats` 等工具

**缺点**:
- 每个成员需要一个额外的 Python 进程（内存开销约 30-50MB）
- 需要进程管理（启动、监控、重启）
- 增加了 wake 脚本的复杂度

**实施工作量**: ~3-4天

---

### 方案 B: `memory-refresh` — 预会话生成器

**原理**: 一个脚本在 `crush run` 之前运行，读取分层记忆数据，应用衰减，将结果写入一个 markdown 文件。crush 直接读取这个文件作为上下文。

```
┌────────────────────────────────────────────────┐
│  wake 脚本 / cron                               │
│  ┌──────────────────┐    ┌──────────────────┐  │
│  │ memory-refresh   │──→│ .ling*/          │  │
│  │ (Python script)  │    │ MEMORY.md        │  │
│  │                  │    │ (精简记忆文本)    │  │
│  │ 读 experience.db │    └──────────────────┘  │
│  │ 读 meta_facts    │           │              │
│  │ 读 shared_facts  │           ↓              │
│  │ 应用衰减         │    ┌──────────────────┐  │
│  └──────────────────┘    │ crush 启动时     │  │
│                          │ 读 MEMORY.md     │  │
│                          │ 作为静态上下文    │  │
│                          └──────────────────┘  │
└────────────────────────────────────────────────┘
```

**MEMORY.md 示例输出**:
```markdown
## 记忆上下文（自动生成，2026-05-07 08:00）

### 相关经验
- [0.8] Python AST解析失败 → 用 ast.parse() 预检，不用正则
- [0.6] SQLite并发写入冲突 → WAL模式 + 重试

### 认知边界
- 擅长: Python编程, PostgreSQL
- 团队共识: 使用TypeScript
```

**优点**:
- 零运行时开销：不需要额外进程
- 极简实现：一个 Python 脚本
- 向后兼容：MEMORY.md 即使过期也有参考价值

**缺点**:
- 无查询感知：生成时不了解即将进行的任务
- 会话期间静态：生成后不更新，直到下次运行
- 衰减不实时：只在运行脚本时计算

**实施工作量**: ~1-2天

### 方案 C: 代理层注入 — 灵通+ 扩展

**原理**: 灵通+ 代理 (127.0.0.1:8765) 在转发请求到 LLM 之前，调用 `build_context_injection()` 并将结果插入系统提示词。

```
┌────────────────────────────────────────────────┐
│  灵通+ proxy (8765)                             │
│  ┌──────────┐   插入记忆上下文   ┌──────────┐  │
│  │ 接收请求  │ ────────────────→ │ 转发到    │  │
│  │          │  (修改system prompt)│ LLM API  │  │
│  └──────────┘                    └──────────┘  │
│       ↑ 识别哪个成员的请求                        │
│       └→ 读取对应的 .ling*/ 记忆数据              │
└────────────────────────────────────────────────┘
```

**优点**:
- 全自动：所有成员无需任何配置变更
- 集中管理：记忆逻辑在一处维护

**缺点**:
- 架构耦合：代理变成"智能中间件"，违背代理作为透明路由层的设计
- 调试困难：注入发生在 crush 和 LLM 之间，用户看不到
- 单点故障：代理崩溃 = 记忆丢失
- 代理复杂度膨胀：灵通+ 本身已有 token 管理、限流等功能

**实施工作量**: ~2-3天（但增加灵通+ 维护负担）

## 3. 推荐方案

### 推荐: 方案 A（MCP 工具服务器）+ 方案 B（预会话生成器）混合实施

**理由**:

1. **方案 A 是正确的长期架构** — MCP 是 crush 唯一的运行时扩展机制，lingbus 已验证此模式可行
2. **方案 B 作为 Phase 1 快速见效** — 在 MCP 服务器开发期间，先用静态生成解决冗长问题
3. **方案 C 被排除** — 代理注入违反关注点分离，增加灵通+ 复杂度

**两阶段实施**:

### Phase 1: 预会话生成器（1-2天）

目标：**立即减少会话遗产冗长度**

1. 编写 `lingclaude/tools/memory_refresh.py`:
   - 接受成员目录路径参数
   - 读取该成员的 `experience.db`、`meta_facts.json`、`shared_facts.json`
   - 调用 `LayeredMemory.build_context_injection()`
   - 应用 `decay()` 衰减
   - 将结果写入 `.ling*/MEMORY.md`（≤500字）
   
2. 集成到 `wake_members.sh`:
   - 在 `crush run` 之前调用 `memory_refresh.py`
   
3. 更新每个成员的 AGENTS.md:
   - 添加 `MEMORY.md` 引用说明

### Phase 2: MCP 工具服务器（3-4天）

目标：**运行时查询感知记忆**

1. 新建 `lingclaude/mcp_memory_server.py`:
   - 基于 MCP 协议实现（参考 lingbus 实现）
   - 6个工具：recall, record, context, meta_get, meta_set, decay
   - 每个实例绑定到成员的 `.ling*/` 目录
   
2. 更新 `crush.json` 全局配置:
   - 添加 `memory-mcp` 服务器配置（端口按成员分配）
   
3. 更新 `wake_members.sh`:
   - 在 `crush run` 之前启动 `memory-mcp` 服务器
   
4. 更新每个成员的 AGENTS.md:
   - 会话开始时调用 `memory_context()`
   - 会话结束前调用 `memory_record()`

### Phase 3: 记忆迁移（1天）

目标：**将现有 handoff.md 中的历史信息迁移到分层记忆**

1. 编写迁移脚本：解析 handoff.md → 提取经验/决策 → 写入 experience.db / meta_facts.json
2. 运行迁移后，handoff.md 仅保留当前会话状态（任务、阻塞项）
3. 预期效果：handoff.md 从平均 2KB 降至 ~500字节
