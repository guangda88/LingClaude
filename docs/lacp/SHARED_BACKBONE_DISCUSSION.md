## 讨论起点

linggit-bot 能否复用灵安主干 -> 灵族能否共用一个主干 -> 7个对外工程已共用lingcore -> 评估极简主义的必要性和可行性

## 核心结论

共用一个抽象（lingmate API）= 灵元意义上的"共用一个主干"。
共用一个实例（一个DB一个进程）= 违反故障隔离，不是极简是冒险。

## 评估摘要

| 维度 | 评分 | 一句话 |
|------|------|--------|
| 必要性 | 7/10 | 灵元一致性要求，减少重复，统一安全 |
| 可行性(API层) | 9/10 | lingmate/lingmemory 已同构 |
| 可行性(实例层) | 3/10 | 进程/DB/故障隔离是硬约束 |
| 时机 | 不是现在 | lingcore 刚落地，灵族还在演化 |

## 关键论点

### 支持

1. 灵元一致性: 2T3A够用，不应该有多个2T3A实现
2. lingcore已证明: 7对外项目共用后 -90%代码量
3. 统一安全: 灵安security_gate天然覆盖所有灵
4. 跨灵协作: linggit-bot读灵信/灵犀/灵忆用同一API

### 反对

1. lingcore共用的是应用层(HTTP框架)，灵族内部业务逻辑不同
2. "多个2T3A实现"不等于"违反灵元" — Unix VFS也是多实现共用接口
3. 过早统一的代价>重复的代价 — "不痛不修"是灵元实践智慧
4. 故障隔离是硬约束 — 灵族48起事故，OOM/restart storm蔓延过

## 三层主干现状

```
应用层: lingcore (7对外项目共用，已落地)
数据层: lingmate/lingmemory (2T3A，灵忆+灵安+linggit-bot共用)
消息层: LingBus pipeline (灵信独立)
```

## 建议路径

1. 现在: lingmate + lingmemory 统一为一个pip包(API层共用，独立DB)
2. 灵安上线后: security_gate作为跨灵统一安全层
3. linggit-bot: 复用灵安主干(MVP验证)
4. 长期: 逐步收敛LingBus/lingxi到lingmate API

每一步都是"先有痛点再减"，不是"为了统一而统一"。

## 灵元视角

灵元说"其小无内，其大无外":
- 其小无内: lingmate 468行已砍到最薄
- 其大无外: 所有灵都在2T3A建模能力之内

共用一个抽象 = 灵元意义上的共用一个主干。
共用一个实例 = 违反故障隔离。
