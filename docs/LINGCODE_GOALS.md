# 灵码(LingCode) — 目标、方向与任务

**更新**: 2026-06-16（会话76）
**状态**: 飞轮已转动（2062条code_trace + 284条coding_rule）

---

## 一、目标

> **做一个能从事件中提取规律、应用规律做编码工作、在灰区不裸奔的编码系统。**

不是做一个强大的模型。当前所有coding model（GPT/DeepSeek/GLM）的模式是：

```
prompt → 模式匹配 → code → 结束
每次从零开始，不积累规律
```

灵码要的是：

```
prompt → query已有规律 → 生成 → 验证 → 从结果提取新规律 → 存下来
                                                ↓
                                        下一次更强的起点
```

### 三个能力，缺一不可

| 能力 | 做什么 | 灵元角色 |
|------|--------|---------|
| **提取规律** | 从编码操作的error→reasoning→fix中提取不变的coding_rule | events→records |
| **应用规律** | 编码时检索适用规律指导生成，不重复试错 | query→指导 |
| **灰区不裸奔** | 需求理解不确定时不猜，escalate确认 | intent_gate中间件 |

三者形成飞轮：规律越多→灰区判断越准→越少返工→越多精力提取规律→escalate到人的机会指数下降。

### 核心约束

> **人类少量参与下的自治。**

不是"机器完全不用人"。是"人只在关键节点参与，其余机器自己跑"。

每escalate到人一次，产生一条新coding_rule。下一次类似的灰区自动处理。escalate到人的概率随着coding_rule积累指数下降。

```
第1轮：100次出入 → 60次灰区 → 40次escalate到人
第10轮：100次出入 → 30次灰区 → 3次escalate到人
第100轮：100次出入 → 5次灰区 → 0.3次escalate到人
```

### 与ARC-AGI-3的启示

ARC-AGI-3最好的AI agent效率只有人类的0.37%。25万次动作 vs 人类几十次。原因不是算力——是AI在积累events（动作）但没从events中提取records（规律）。

灵码的核心命题：**能力 = 从事件中抽取规律 × 应用这些规律。不是参数量，不是训练数据量，是抽象层次。**

## 二、方向

### 灵元薄主干

```
意图 → [intent_gate: 灰区确认] → 生成 → 验证 → [提取规律] → [存规律] → 反馈
                                                                    ↓
                                                           query规律指导下次
```

这是主干。永不变。

模型是插片。RAG源是插片。评估标准是插片。路由策略是插片。

### 数据飞轮：一个飞轮，两个入口

```
     离线入口                          在线入口
  crush.db历史挖掘                 编码时hook自动采集
       ↓                               ↓
  code_trace                      code_trace
       ↓                               ↓
  coding_rule ←←← 同一个灵忆 ←←← coding_rule
       ↓
  query → 指导下一次编码
```

- **离线**：从全族20万条crush.db消息中挖掘error→reasoning→fix模式
- **在线**：灵克编码链路接入DataFlywheel，每次edit→test→fix自动采集+提取

### 三个核心type

| type | 角色 | 当前数量 |
|------|------|---------|
| code_trace | 编码轨迹（events素材） | 2,062 |
| coding_rule | 编码规律（records知识） | 284 |
| intent_gate | 意图澄清（灰区防护） | 待实现 |

### 小模型+知识底座架构

灵码不依赖大GPU。目标是**核显或CPU能运行的小模型(1-3B) + 灵忆DB(编码规律) + 2T3A(查询接口) = 强编码能力**。

```
用户需求
  ↓
1-3B小模型理解意图（核显推理，不需要大GPU）
  ↓
query(type=coding_rule, state=validated/generalized) → 拿到适用规律
  ↓
query(type=code_trace, prompt=相似任务) → 拿到历史代码范本
  ↓
小模型组装规则+范本 → 生成代码（核显推理）
  ↓
验证通过 → 存为新trace → 若发现新pattern → 存为新rule
  ↓
验证失败 → query更底层的规律 → 仍然不走大模型
  ↓
如果真的从未见过（查表命中率<30%）→ 调大模型蒸馏新rule → 下次不调了
```

**能查表解决的问题不要推理。** 灵忆DB中累计的228条rule越多，查表命中率越高，调大模型的频率越低。

| 规模 | 参数 | 硬件 | 推理方式 | 覆盖场景 |
|------|------|------|---------|---------|
| 小模型 | 1-3B | CPU/核显 | 查表+组装 | 80%日常编码 |
| 中模型 | 7-14B | 老GPU(RTX 2060) | 查表+少量推理 | 15%中等复杂 |
| 大模型 | ∞ | 云GPU(A100) | 完全推理 | 5%从未见过的问题 → 蒸馏后变查表 |

随着时光推移，5%会越来越趋近0%。**灵码的算力需求是随时间递减的。**

这不仅是技术选择，更是V1.0的应用——参数量是插片，知识底座才是主干。

## 五、灵码薄主干

```
意图 → [intent_gate] → 生成 → 验证 → [提取规律] → [存规律] → 反馈
         灰区不裸奔              events→records    query指导
```

code_trace是events（血液），coding_rule是records（骨骼），intent_gate是免疫层（灰区防护）。

灵码飞轮需要三者——血液流转不息，骨骼支撑结构，免疫层防止方向性错误。
